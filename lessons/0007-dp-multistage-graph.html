Lesson 7 · Ch7 Dynamic Programming

Multistage Graphs: Forward & Backward

Exam goal: hand-trace a shortest path through a staged graph both ways and write every min{…} line. The single win: see why the recurrence beats greedy, and be able to reproduce d(S,T)=9 from the same figure forward and backward without notes.

0 · What is this all about?

Dynamic programming (DP) is a way to solve a problem by viewing it as a sequence of decisions. The trick: break the problem into smaller subproblems, solve each one once, store the answer, and reuse it instead of recomputing it. The greedy method (the rival) just grabs the locally cheapest choice at each step and never looks back. DP is smarter: it does not commit to a choice that looks cheap right now — it weighs how that choice pays off later. That is why DP can find the true best answer where greedy gets fooled.

Everything here lives on a graph. A few words, in plain terms:

  • A graph is just dots connected by lines.
  • Each dot is a vertex (also called a node).
  • Each line is an edge connecting two vertices.
  • A directed edge is a one-way arrow: u→v lets you go from u to v but not back.
  • Each edge has a number, its weight or cost. A path is a chain of edges; its path cost is the sum of the weights along it.

A multistage graph is a special directed graph. Its vertices are grouped into ordered stages (stage 1, stage 2, …), and every edge only goes from one stage to the very next one — never sideways, never skipping ahead. There is exactly one start vertex (in the first stage) and one end vertex (in the last stage). The goal: find the cheapest path from start to end.

stage 1 stage 2 stage 3 stage 4 ┌── A ──┐ ┌── D ──┐ S ─────────┼── B ──┼────┼── E ──┼──── T └── C ──┘ └── F ──┘ (start) (end)

Our running example has 4 stages: S alone in stage 1; A, B, C in stage 2; D, E, F in stage 3; T alone in stage 4. We want the cheapest way to get from S to T. The exact edge weights appear in §2; the rest of the lesson is about computing that cheapest path the DP way (and seeing why greedy can't).

Intuition to hold onto: picking each step by "what's cheapest right now" can trap you. DP instead asks "from this vertex, what's the cheapest way to finish?" — solving small finish-from-here subproblems once and reusing them. That reuse is the whole point.

1 · Why greedy fails here

Greedy "take the cheapest outgoing edge each stage" looks smart and is wrong. From S it grabs S→A (cost 1), then A→D (4), then D→T (18):

greedy: S →A→D→T = 1 + 4 + 18 = 23 optimum: S →C→F→T = 5 + 2 + 2 = 9

The true optimum starts with the most expensive-looking first edge, S→C = 5 — because C opens onto C→F=2 and F→T=2, dirt cheap downstream. A locally bad choice that pays off later: that is exactly the situation greedy cannot see and dynamic programming can.

2 · Multistage graph — the formal definition

Now the precise version of the idea from §0. A multistage graph G=(V,E) is a directed graph whose vertices split into k≥2 disjoint stages V₁,…,V_k, where every edge ⟨u,v⟩ jumps from one stage to the next (u∈Vᵢ, v∈V_{i+1}), and the first and last stages are single vertices: |V₁|=|V_k|=1. The lone source (start) is s, the lone sink (end) is t. The problem: find the minimum-cost s→t path.

Our running figure has 4 stages, with the verified edge weights below.

3 · Backward reasoning — trace it yourself

Let d(X,T) = cost of the shortest path from X to the sink T. Start at the stage just before T (the base cases d(D,T)=18, d(E,T)=13, d(F,T)=2) and work backward toward S. Compute d(C,T), d(A,T), d(B,T), then d(S,T) on paper before opening the answer.

Backward computation (every min line)
d(C,T) = min{ 2 + d(F,T) } = 2 + 2 = 4 d(A,T) = min{ 4 + d(D,T), 11 + d(E,T) } = min{ 4 + 18, 11 + 13 } = min{ 22, 24 } = 22 d(B,T) = min{ 9 + d(D,T), 5 + d(E,T), 16 + d(F,T) } = min{ 9 + 18, 5 + 13, 16 + 2 } = min{ 27, 18, 18 } = 18 d(S,T) = min{ 1 + d(A,T), 2 + d(B,T), 5 + d(C,T) } = min{ 1 + 22, 2 + 18, 5 + 4 } = min{ 23, 20, 9 } = 9

Answer: d(S,T)=9, achieved by S→C→F→T. Each vertex's optimum is written in terms of vertices closer to the sink, then resolved from the sink back — that is backward reasoning.

4 · Forward reasoning — same graph, other direction

Now let d(S,X) = cost of the shortest path from S to X. Start at the stage just after S and work forward. Base cases are the direct S-edges; then stage 3, then T.

d(S,A) = 1 d(S,B) = 2 d(S,C) = 5 d(S,D) = min{ d(S,A)+4, d(S,B)+9 } = min{ 1+4, 2+9 } = min{ 5, 11 } = 5 d(S,E) = min{ d(S,A)+11, d(S,B)+5 } = min{ 1+11, 2+5 } = min{ 12, 7 } = 7 d(S,F) = min{ d(S,B)+16, d(S,C)+2 } = min{ 2+16, 5+2 } = min{ 18, 7 } = 7 d(S,T) = min{ d(S,D)+18, d(S,E)+13, d(S,F)+2 } = min{ 5+18, 7+13, 7+2 } = min{ 23, 20, 9 } = 9

Same optimum d(S,T)=9, reached by expressing each vertex's cost via vertices closer to the source. Two directions, one answer — the recurrence is consistent either way.

Key exam insight: forward and backward are both correct and give the same value. They differ only in what the subproblem is: backward stores "cost from X to sink," forward stores "cost from source to X." Pick whichever the question's figure makes easier to fill — and always write the min{…} lines, that is where the marks are.

5 · Principle of optimality

Statement. If a problem is solved by a sequence of decisions D₁,…,Dₙ and that sequence is optimal, then for any k the last k decisions must themselves be optimal (an optimal solution is built from optimal sub-solutions).

Shortest-path corollary. If i, i₁, i₂, …, j is a shortest path from i to j, then the suffix i₁, i₂, …, j must be a shortest path from i₁ to j. (If it weren't, swap in a cheaper suffix and the whole path gets cheaper — contradicting that it was shortest.)

This corollary is what licenses the recurrence d(X,T)=min_v{w(X,v)+d(v,T)}: because every optimal suffix is itself optimal, we may reuse the already-computed d(v,T) instead of re-searching. Consequence: any problem expressible as a multistage graph can be solved by dynamic programming.
