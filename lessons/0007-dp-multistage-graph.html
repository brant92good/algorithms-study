0 · What is this all about?0 · 這課到底在幹嘛？

Dynamic programming (DP) is a way to solve a problem by viewing it as a sequence of decisions. The trick: break the problem into smaller subproblems, solve each one once, store the answer, and reuse it instead of recomputing it. The greedy method (the rival) just grabs the locally cheapest choice at each step and never looks back. DP is smarter: it does not commit to a choice that looks cheap right now — it weighs how that choice pays off later. That is why DP can find the true best answer where greedy gets fooled.Dynamic programming（DP）的想法，就是把一個問題看成一連串的決策（sequence of decisions）來解。重點在於：把大問題切成一個個小 subproblem，每個只算一次，算完把答案存起來，下次要用直接拿，不要重算。它的對手 greedy 則是每一步都挑當下最便宜的，挑了就走、從不回頭。DP 聰明在哪？它不會看到眼前便宜就急著選，而是會去想：這步現在划算，之後會不會反而虧？所以 greedy 被騙的地方，DP 還是能找到真正的最佳解。

Everything here lives on a graph. A few words, in plain terms:這一切都發生在一張 graph 上。幾個名詞先用白話講清楚：

  • A graph is just dots connected by lines.graph 就是一堆點，用線連起來而已。
  • Each dot is a vertex (also called a node).每個點叫一個 vertex（也叫 node）。
  • Each line is an edge connecting two vertices.每條線是一條 edge，把兩個 vertex 接起來。
  • A directed edge is a one-way arrow: u→v lets you go from u to v but not back.directed edge 就是單向箭頭：u→v 只能從 u 走到 v，不能倒回來。
  • Each edge has a number, its weight or cost. A path is a chain of edges; its path cost is the sum of the weights along it.每條 edge 上面有一個數字，就是它的 weight（權重）、也就是成本。一條 path 是一串接起來的 edge；它的 path cost 就是把沿路的 weight 全部加起來。

A multistage graph is a special directed graph. Its vertices are grouped into ordered stages (stage 1, stage 2, …), and every edge only goes from one stage to the very next one — never sideways, never skipping ahead. There is exactly one start vertex (in the first stage) and one end vertex (in the last stage). The goal: find the cheapest path from start to end.multistage graph 是一種特別的有向圖。它的 vertex 被排成一層一層有順序的 stage（stage 1、stage 2、…），而且每條 edge 都只能從某一層走到緊接著的下一層——不能橫著走、也不能跳級。第一層只有一個 start vertex，最後一層只有一個 end vertex。要做的事很單純：找從 start 到 end 最便宜的那條 path。

stage 1 stage 2 stage 3 stage 4 ┌── A ──┐ ┌── D ──┐ S ─────────┼── B ──┼────┼── E ──┼──── T └── C ──┘ └── F ──┘ (start) (end)

Our running example has 4 stages: S alone in stage 1; A, B, C in stage 2; D, E, F in stage 3; T alone in stage 4. We want the cheapest way to get from S to T. The exact edge weights appear in §2; the rest of the lesson is about computing that cheapest path the DP way (and seeing why greedy can't).我們整課會一直用同一個範例，它有 4 個 stage：stage 1 只有 S；stage 2 是 A, B, C；stage 3 是 D, E, F；stage 4 只有 T。目標是找 S 到 T 最便宜的走法。實際的 edge weight 留到 §2 再給；接下來整課就是在示範怎麼用 DP 把這條最便宜的 path 算出來（順便看 greedy 為什麼做不到）。

Intuition to hold onto: picking each step by "what's cheapest right now" can trap you. DP instead asks "from this vertex, what's the cheapest way to finish?" — solving small finish-from-here subproblems once and reusing them. That reuse is the whole point.記住這個直覺：每一步都只看「現在哪個最便宜」很容易踩雷。DP 換個問法——「從這個 vertex 出發，走到終點最便宜要花多少？」把這種「從這裡走完剩下的路」的小 subproblem 算一次，之後一直重複用。會重複用，就是整個 DP 的重點。

1 · Why greedy fails here1 · greedy 在這裡為什麼會翻車

Greedy "take the cheapest outgoing edge each stage" looks smart and is wrong. From S it grabs S→A (cost 1), then A→D (4), then D→T (18):Greedy 的策略是「每個 stage 都挑最便宜的出邊」，看起來很合理，但其實是錯的。從 S 開始，它會先抓 S→A（成本 1），再抓 A→D（4），最後 D→T（18）：

greedy: S →A→D→T = 1 + 4 + 18 = 23 optimum: S →C→F→T = 5 + 2 + 2 = 9

The true optimum starts with the most expensive-looking first edge, S→C = 5 — because C opens onto C→F=2 and F→T=2, dirt cheap downstream. A locally bad choice that pays off later: that is exactly the situation greedy cannot see and dynamic programming can.真正的最佳解偏偏是從看起來最貴的第一條 edge 起手，S→C = 5——因為走 C 之後會接到 C→F=2、F→T=2，後面便宜到不行。第一步先吃點虧、後面整個賺回來：這種狀況 greedy 完全看不出來，但 dynamic programming 看得到。

2 · Multistage graph — the formal definition2 · Multistage graph——正式定義

Now the precise version of the idea from §0. A multistage graph G=(V,E) is a directed graph whose vertices split into k≥2 disjoint stages V₁,…,V_k, where every edge ⟨u,v⟩ jumps from one stage to the next (u∈Vᵢ, v∈V_{i+1}), and the first and last stages are single vertices: |V₁|=|V_k|=1. The lone source (start) is s, the lone sink (end) is t. The problem: find the minimum-cost s→t path.把 §0 講的那套寫成精確版本。一張 multistage graph G=(V,E) 是有向圖，它的 vertex 被分成 k≥2 個互不重疊的 stage V₁,…,V_k，每條 edge ⟨u,v⟩ 都是從某一層跳到下一層（u∈Vᵢ, v∈V_{i+1}），而且頭尾兩層各只有一個 vertex：|V₁|=|V_k|=1。那個唯一的起點 source 叫 s，唯一的終點 sink 叫 t。要解的問題就是：找成本最小的 s→t path。

3 · Backward reasoning — trace it yourself3 · 反向（backward）推理——自己動手推

Let d(X,T) = cost of the shortest path from X to the sink T. Start at the stage just before T (the base cases d(D,T)=18, d(E,T)=13, d(F,T)=2) and work backward toward S. Compute d(C,T), d(A,T), d(B,T), then d(S,T) on paper before opening the answer.先定義 d(X,T) = 從 X 走到 sink T 的最短路徑成本。從 T 前一層開始（base case 是 d(D,T)=18, d(E,T)=13, d(F,T)=2），一路往回推到 S。建議你先在紙上把 d(C,T), d(A,T), d(B,T) 算出來，再算 d(S,T)，算完再打開答案對。

Backward computation (every min line)反向計算（每一條 min 都列出來）
d(C,T) = min{ 2 + d(F,T) } = 2 + 2 = 4 d(A,T) = min{ 4 + d(D,T), 11 + d(E,T) } = min{ 4 + 18, 11 + 13 } = min{ 22, 24 } = 22 d(B,T) = min{ 9 + d(D,T), 5 + d(E,T), 16 + d(F,T) } = min{ 9 + 18, 5 + 13, 16 + 2 } = min{ 27, 18, 18 } = 18 d(S,T) = min{ 1 + d(A,T), 2 + d(B,T), 5 + d(C,T) } = min{ 1 + 22, 2 + 18, 5 + 4 } = min{ 23, 20, 9 } = 9

Answer: d(S,T)=9, achieved by S→C→F→T. Each vertex's optimum is written in terms of vertices closer to the sink, then resolved from the sink back — that is backward reasoning.答案：d(S,T)=9，走的是 S→C→F→T。注意每個 vertex 的最佳值，都是用更靠近 sink 的 vertex 來表示，再從 sink 一路往回解——這套就叫 backward reasoning（反向推理）。

4 · Forward reasoning — same graph, other direction4 · 正向（forward）推理——同一張圖，反過來推

Now let d(S,X) = cost of the shortest path from S to X. Start at the stage just after S and work forward. Base cases are the direct S-edges; then stage 3, then T.這次改定義 d(S,X) = 從 S 走到 X 的最短路徑成本。從 S 後一層開始，一路往前推。base case 就是 S 直接連出去的那幾條邊；接著推 stage 3，最後推到 T。

d(S,A) = 1 d(S,B) = 2 d(S,C) = 5 d(S,D) = min{ d(S,A)+4, d(S,B)+9 } = min{ 1+4, 2+9 } = min{ 5, 11 } = 5 d(S,E) = min{ d(S,A)+11, d(S,B)+5 } = min{ 1+11, 2+5 } = min{ 12, 7 } = 7 d(S,F) = min{ d(S,B)+16, d(S,C)+2 } = min{ 2+16, 5+2 } = min{ 18, 7 } = 7 d(S,T) = min{ d(S,D)+18, d(S,E)+13, d(S,F)+2 } = min{ 5+18, 7+13, 7+2 } = min{ 23, 20, 9 } = 9

Same optimum d(S,T)=9, reached by expressing each vertex's cost via vertices closer to the source. Two directions, one answer — the recurrence is consistent either way.一樣得到最佳解 d(S,T)=9，只是這次每個 vertex 的成本改用更靠近 source 的 vertex 來表示。兩個方向推、答案一模一樣——這條遞迴式不管往哪邊推都一致。

Key exam insight: forward and backward are both correct and give the same value. They differ only in what the subproblem is: backward stores "cost from X to sink," forward stores "cost from source to X." Pick whichever the question's figure makes easier to fill — and always write the min{…} lines, that is where the marks are.考試重點：forward 跟 backward 兩個都對，算出來的值也一樣。差別只在subproblem 的定義不同：backward 存「從 X 到 sink 的成本」，forward 存「從 source 到 X 的成本」。看題目給的圖哪個方向比較好填，就挑哪個——重點是 min{…} 那幾行一定要寫出來，分數就是給在那裡。

5 · Principle of optimality5 · Principle of optimality（最佳化原則）

Statement. If a problem is solved by a sequence of decisions D₁,…,Dₙ and that sequence is optimal, then for any k the last k decisions must themselves be optimal (an optimal solution is built from optimal sub-solutions).敘述。假設一個問題是靠一連串決策 D₁,…,Dₙ 解掉的，而且這一整串是最佳的，那麼不管挑哪個 k，最後 k 個決策本身一定也是最佳的（簡單講：最佳解是用一堆最佳子解拼出來的）。

Shortest-path corollary. If i, i₁, i₂, …, j is a shortest path from i to j, then the suffix i₁, i₂, …, j must be a shortest path from i₁ to j. (If it weren't, swap in a cheaper suffix and the whole path gets cheaper — contradicting that it was shortest.)套到最短路徑上。如果 i, i₁, i₂, …, j 是 i 到 j 的最短路徑，那它的後半段 i₁, i₂, …, j 一定也是 i₁ 到 j 的最短路徑。（用反證想：要是後半段不是最短，那把它換成更便宜的一段，整條 path 就更便宜了——但這跟「它本來就是最短」矛盾。）

This corollary is what licenses the recurrence d(X,T)=min_v{w(X,v)+d(v,T)}: because every optimal suffix is itself optimal, we may reuse the already-computed d(v,T) instead of re-searching. Consequence: any problem expressible as a multistage graph can be solved by dynamic programming.就是這個推論撐起了遞迴式 d(X,T)=min_v{w(X,v)+d(v,T)}：既然每段最佳後段本身也是最佳的，那我們就能直接拿已經算好的 d(v,T) 來用，不用重新再搜一次。所以：只要一個問題能畫成 multistage graph，就能用 dynamic programming 解。
