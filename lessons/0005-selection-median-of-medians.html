Lesson 5 · Prune-and-Search

Linear-Time Selection

Exam goal: hand-trace the 3-way partition to find a k-th smallest element, and recite why groups of 5 give O(n). Sorting finds the median in O(n log n) — this lesson's single tangible win is finding it in O(n), by never sorting the whole set, only pruning a guaranteed fraction each round.

1 · What are we even solving?

The selection problem: you're given a set of n numbers (in no particular order) and a rank k, and you must return the k-th smallest element — the value that would sit in position k if you sorted the set, but found without fully sorting. The median is the middle one: the special case k ≈ n/2 (precisely the ⌈n/2⌉-th smallest).

S = {7, 2, 9, 4, 5} (n = 5) sorted → 2 4 5 7 9 ← we never actually do this 1 2 3 4 5 ← rank k k = 1 → 2 (smallest) k = 3 → 5 (the median) k = 5 → 9 (largest)

The obvious recipe is: sort the whole set, then read off position k. Sorting costs O(n log n). But we only want one element, not the whole order — so fully sorting is wasteful. The single win of this lesson is finding the answer in O(n) (linear — work proportional to n, no log n factor), by never sorting the whole set.

The idea behind that win — prune-and-search. Think of how binary search finds a value in a sorted array: each step throws away half the array (the half that can't contain the target) and repeats on the rest. Prune-and-search generalizes this to unsorted data: each round, prove that some constant fraction of the elements cannot be the answer, throw them away ("prune"), and repeat on what survives. Because each round discards a fixed slice, the survivors shrink geometrically and the total work stays small.

To prune, we need to split the set. We pick one element as the pivot p (a reference value we compare everything against) and do a partition — one linear sweep that drops every element into one of three buckets: those smaller than p, those equal to p, and those larger than p. Knowing how many fall below p tells us immediately which bucket the k-th smallest lives in, so we can discard the other two. Sections 2–3 make this precise; sections 4–5 show how to choose p so the pruning is guaranteed.

2 · The prune-and-search method

Binary search runs in many iterations; each one throws away a fixed fraction f of the data and recurses on the rest. Prune-and-search generalizes that: prune a fraction f, recurse on the remaining (1−f)·n elements, until the survivor is small enough to solve directly. If each iteration's work is O(nᵏ), then:

T(n) = T((1 − f)·n) + O(nᵏ) ⇒ T(n) = O(nᵏ)
Why the first iteration dominates: unrolling gives c·nᵏ·(1 + (1−f)ᵏ + (1−f)²ᵏ + …). Since 1−f < 1, that bracket is a convergent geometric series — a constant. So the total cost is just a constant times the first iteration's nᵏ. The whole process is the same order as one pass.

3 · The 3-way partition

Pick a pivot p and split S into three buckets (the partition), then recurse into only one. |S₁| means "the number of elements in S₁".

BucketHolds
S₁everything < p
S₂everything = p
S₃everything > p

To find the k-th smallest, compare k against the bucket sizes:

if |S₁| ≥ k → answer is in S₁; recurse on S₁ with same k else if |S₁|+|S₂| ≥ k → answer is p itself; DONE else → answer is the (k − |S₁| − |S₂|)-th smallest of S₃; recurse on S₃

4 · Trace it yourself

Let S = {7, 2, 9, 4, 5, 1, 8, 3} (n = 8). Find the 4th smallest (k = 4). Suppose this round's pivot is p = 5. Work the partition and the branch on paper, then reveal.

Step 1 — build the three buckets around p = 5

S₁ = {2, 4, 1, 3} (the four elements < 5)  |  S₂ = {5}  |  S₃ = {7, 9, 8}.
So |S₁| = 4, |S₂| = 1, |S₃| = 3.

Step 2 — which bucket do we recurse into?

Test |S₁| ≥ k → 4 ≥ 4 ✓. The 4th smallest lies in S₁. Prune S₂ and S₃ and recurse on S₁ = {2, 4, 1, 3} with the same k = 4.
(Sorted, S₁ is 1,2,3,4 — its 4th smallest is 4, which is indeed the 4th smallest of the original set. ✓)

What if instead k = 5?

|S₁| ≥ 5? 4 ≥ 5 ✗. Then |S₁|+|S₂| ≥ 5? 5 ≥ 5 ✓ → the answer is p = 5 itself. Done, no recursion.
And k = 6? Both tests fail, so recurse into S₃ for the (6 − 4 − 1) = 1st smallest of {7,9,8}, which is 7. ✓

5 · Median of medians — the pivot trick

A bad pivot prunes almost nothing (worst case O(n²)). We need a pivot guaranteed "central enough" regardless of input:

1. Split S into ⌈n/5⌉ groups of 5. 2. Sort each group of 5; its median is the 3rd element. 3. p = the median of those ⌈n/5⌉ group-medians.

The grid below is slide 7. Each column is one sorted group of 5 (smallest on top). The middle row M is the group-medians; P is their median.

sorted ↓ each column = one sorted 5-element group ┌───────────────────────────┐ · │ · · · · │ · · · · │ · · · · │ · · · M → · · [·]──P · · │ · · · ← median row M · · · · │ · · · · · · · · │ · · · · └────────────────────┘ upper-left block: ≥ ¼ of S is ≤ P lower-right block: ≥ ¼ of S is ≥ P
The guarantee: about half the groups have median ≤ P; in each such group the median plus the two above it (3 of 5) are ≤ P. That forces at least n/4 elements ≤ P, and symmetrically at least n/4 ≥ P. So at least n/4 are pruned every iteration and at most 3n/4 survive.

6 · Why it's O(n)

The 5-step algorithm: (1) group by 5, (2) sort each group, (3) recursively find the median of the n/5 medians = the pivot p, (4) partition into S₁,S₂,S₃ around p, (5) the 3-way recurse on at most 3n/4 survivors. Steps 1, 2, 4 are each O(n); step 3 costs T(n/5); step 5 costs T(3n/4). Hence:

T(n) = T(3n/4) + T(n/5) + O(n)

The two fractions sum to 3/4 + 1/5 = 19/20 < 1 — that slack under 1 is exactly why this stays linear.

The substitution proof (algebra)

Assume T is bounded by a polynomial. Comparing coefficients of the two recursive terms against T(19n/20): the linear parts match (3/4 + 1/5 = 19/20) and the quadratic part of the sum is smaller (9/16 + 1/25 = 241/400 < 361/400). So T(3n/4) + T(n/5) ≤ a₀ + T(19n/20), collapsing the recurrence to a single term:

T(n) ≤ cn + T(19n/20) ≤ cn·(1 + 19/20 + (19/20)² + …) = cn · 1/(1 − 19/20) = cn · 20 = 20cn ∴ T(n) ≤ 20cn + b = O(n)

The geometric series with ratio 19/20 sums to 20. The constant 20 is fixed, while the sort's log n grows without bound — that's the asymptotic win.
