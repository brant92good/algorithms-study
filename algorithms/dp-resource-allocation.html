Dynamic Programming · Ch7

Resource Allocation (DP)

You have a fixed pile of resources and several projects competing for them. Hand them out to maximize total profit — and let dynamic programming try every split for you.

📄 Source: Alg07 · Ch 7 §9 (Resource Allocation), slides 19–21.

What is it?

You have m identical units of some resource (money, machines, staff — anything divisible into whole units) and n projects. Giving a project more units earns more profit, but the units are limited, so you must choose how many to give each project. The goal: pick an allocation that maximizes the sum of profits across all projects.

The profit of each choice is given by a table p(i, j) = the profit you get when you allocate j units to project i. (Giving 0 units to a project earns 0.) Here is the deck's instance — m = 3 units, n = 4 projects:

p(i,j)j=1j=2j=3
1289
2567
3444
4245

So p(1,2) = 8 means "2 units to project 1 earns 8 profit." Brute force would try every way to split 3 units among 4 projects — wasteful. DP instead asks a smaller question, project by project: "using only the first i projects and q units of budget, what is the best profit?" Call that f(i, q). The key insight (the decision): when we add project i, we decide how many units k to hand it (0, 1, … up to q), spend the rest on the earlier projects, and keep the best:

f(i, q) = max over k = 0..q of p(i, k) + f(i-1, q - k)

with base row f(1, q) = p(1, q). The answer is f(n, m) = f(4, 3). (The deck draws the same thing as a multistage graph: a node (i, j) = "i units used by the first j projects", edges weighted by profit increments, and the answer is the longest / max-profit path from S to T.)

Watch it run

We fill the DP table f(i, q) row by row. Row 1 just copies project 1's profits. Each later cell is the max over "how many units k to give project i" — gold is the cell being filled, blue cells are the earlier-row values it depends on. Total budget m = 3, so the answer lives in the bottom-right cell.

Type: DP Time: O(n · m²) Answer: profit 13
