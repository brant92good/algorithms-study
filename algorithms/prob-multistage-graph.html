Multistage Graph Shortest PathMultistage Graph Shortest Path

Find the cheapest route across a graph whose nodes come in ordered stages — a problem where grabbing the cheapest first edge gets you burned, and the clean first taste of dynamic programming.在一張節點分成好幾個 stage（階段）的圖上，找出最便宜的路線——這題裡，貪心地抓第一條最便宜的邊反而會踩雷，也是 dynamic programming 最乾淨的入門範例。

The problem這題在問什麼

A multistage graph is a directed graph whose nodes are partitioned into ordered groups called stages. Every edge points from one stage to the very next stage — you can never skip a stage or go backward. The first stage is a single source node S; the last stage is a single sink node T. Each edge carries a cost (its weight).multistage graph 是一張有向圖，節點被分成一組組有先後順序的 stage。每條邊都從某個 stage 指向下一個 stage——你不能跳過 stage，也不能往回走。第一個 stage 只有一個 source 節點 S；最後一個 stage 只有一個 sink 節點 T。每條邊都帶一個成本（它的 weight）。

What's asked: find the minimum-cost path from S to T — the sequence of edges, one per stage transition, whose weights add up to the smallest total.要求的是：從 S 到 T 的最小成本路徑——一連串的邊（每跨一個 stage 一條），讓 weight 加起來總和最小。

The concrete instance (our running example). The graph has 4 stages: S → then {A, B, C} → then {D, E, F} → then T. So a path is three decisions in a row: which stage-2 node, which stage-3 node, then on to T. Every edge weight:具體例子（我們一路用的範例）。這張圖有 4 個 stage：S → {A, B, C} → {D, E, F} → T。所以一條路徑就是連續三個決定：選哪個 stage-2 節點、選哪個 stage-3 節點，再到 T。每條邊的 weight：

  • Stage 1→2: S→A=1, S→B=2, S→C=5
  • Stage 2→3: A→D=4, A→E=11, B→D=9, B→E=5, B→F=16, C→F=2
  • Stage 3→4: D→T=18, E→T=13, F→T=2

Why is this hard? Why not just be greedy? The greedy method takes the cheapest edge visible right now. From S that's S→A (cost 1), then A→D (cost 4), then the forced D→T (cost 18) — total 23. But the true cheapest route is S→C→F→T = 5+2+2 = 9. The first edge S→C looks expensive (cost 5), yet it unlocks a dirt-cheap finish. A locally cheap choice can trap you in an expensive future — greedy can't see that far ahead, so we need a method that reasons about the whole path.這題難在哪？為什麼不能直接 greedy？greedy 每一步都挑當下看得到最便宜的邊。從 S 出發就是 S→A（成本 1），接著 A→D（成本 4），然後被迫走 D→T（成本 18）——總共 23。但真正最便宜的路是 S→C→F→T = 5+2+2 = 9。第一條邊 S→C 看起來很貴（成本 5），卻接上了超便宜的後半段。一個當下便宜的選擇，可能害你卡進一個很貴的未來——greedy 看不了那麼遠，所以我們需要一個會考慮整條路的方法。

Watch it solved看它怎麼解

The solving idea is backward reasoning. Let d(X,T) = the cost of the cheapest path from node X to the sink T. Start at the nodes nearest T (their answer is just the single edge to T), then step one stage back at a time: for each node take "edge to a neighbor + that neighbor's already-known d(·,T), keep the smallest." The graph is below with every weight shown; we fill each node's d(·,T) badge working backward from T. Green = its d(·,T) is known; amber = the node being computed this step. At the end the optimal path edges turn solid.解法的核心是 backward reasoning（從後往前推）。令 d(X,T) = 從節點 X 到 sink T 的最便宜成本。先從離 T 最近的節點開始（它們的答案就是那條直接到 T 的邊），再一個 stage 一個 stage 往回推：每個節點就算「到某個鄰居的邊 + 那個鄰居已經算好的 d(·,T)，取最小的」。下面是整張圖，每條 weight 都標出來；我們從 T 往回幫每個節點填上 d(·,T) 標籤。綠色 = 它的 d(·,T) 已經知道了；琥珀色 = 這一步正在算的節點。最後最佳路徑的邊會變成實線。

Why is the order C, then A, then B? It isn't — A, B, C are all stage-2 nodes, and within a single stage the order is arbitrary. Once every stage-3 node (D, E, F) has its d(·,T) known, any stage-2 node can be computed; their answers don't depend on each other. The animation just happens to show C first, then A, then B — computing them in any order gives the exact same badges.為什麼順序是 C、再 A、再 B？其實沒有特別原因——A、B、C 都是 stage-2 的節點，同一個 stage 裡面誰先誰後都沒差。只要 stage-3 的節點（D、E、F）的 d(·,T) 都算好了，任何一個 stage-2 節點就都能算了；它們彼此不互相依賴。動畫只是剛好先顯示 C、再 A、再 B 而已——換成任何順序去算，填出來的標籤都一模一樣。

Method: Dynamic Programming Time: O(V + E) Answer: d(S,T) = 9 Beats greedy: 9 vs 23

Where does O(V+E) come from? V is the number of nodes (vertices) and E the number of edges. We compute d(·,T) for each node exactly once — that's the V part. And at each node we scan its outgoing edges once (each edge is looked at a single time, when we relax the node it leaves from) — across the whole graph that's E edge-touches total. So the work is proportional to the number of nodes plus the number of edges: O(V+E). In our example that's 8 nodes and 12 edges — a tiny, single linear sweep.O(V+E) 是怎麼來的？V 是節點數，E 是邊數。每個節點的 d(·,T) 我們只算一次——這就是 V 那部分。而在每個節點上，我們把它的出邊各掃一次（每條邊只會被看一次，就是在處理它的起點節點時）——整張圖加起來總共就是 E 次的看邊。所以總工作量正比於節點數加邊數：O(V+E)。以我們的例子來說就是 8 個節點、12 條邊——一趟很小的線性掃過去而已。

The other direction — forward reasoning另一個方向——forward reasoning

The animation used backward reasoning: d(X,T) = cheapest cost from X to the sink, filled in from T back toward S — giving d(C,T)=4, d(A,T)=22, d(B,T)=18, and finally d(S,T)=9. The very same graph can be solved the opposite way. In forward reasoning you define d(S,X) = cheapest cost from the source S down to node X, and fill it in from S forward toward T. The stage-2 nodes are just their single edge from S (d(S,A)=1, d(S,B)=2, d(S,C)=5); then for each later node you take "each incoming neighbor's d(S,·) plus the edge into this node, keep the smallest." That gives d(S,D)=5, d(S,E)=7, d(S,F)=7, and finally d(S,T)=min{5+18, 7+13, 7+2}=9 — the same optimum of 9, reached from the front instead of the back.動畫用的是 backward reasoning：d(X,T) = 從 X 到 sink 的最便宜成本，從 T 往 S 倒著填——得到 d(C,T)=4、d(A,T)=22、d(B,T)=18，最後 d(S,T)=9。同一張圖也可以反過來做。forward reasoning 則是定義 d(S,X) = 從 source S 到節點 X 的最便宜成本，從 S 往 T 順著填。stage-2 的節點就是那條從 S 來的邊（d(S,A)=1、d(S,B)=2、d(S,C)=5）；之後每個節點就算「每個進來的鄰居的 d(S,·) 加上進到這個節點的那條邊，取最小」。這樣得到 d(S,D)=5、d(S,E)=7、d(S,F)=7，最後 d(S,T)=min{5+18, 7+13, 7+2}=9——一樣的最佳解 9，只是從前面推過來而已。

Principle of optimality. Both directions work for the same reason: if a sequence of decisions is optimal overall, then any tail (or head) of that sequence must itself be optimal — so an optimal path is built from optimal sub-paths, which is exactly what lets us reuse the stored d(·) values instead of re-exploring. Concretely: d(C,T)=4 is computed once and then reused for every path that passes through C (and there's only one here, S→C) — it is never recomputed. That re-use is the whole point of DP: a node's cheapest-to-T cost doesn't change based on how you arrived, so storing it once and looking it up beats recalculating it for each path.Principle of optimality（最佳化原理）。兩個方向能成立的理由是同一個：如果一整串決定整體是最佳的，那它的任何一段尾巴（或開頭）本身也一定是最佳的——所以最佳路徑是由最佳的子路徑組成的，這正是為什麼我們可以重複利用存起來的 d(·) 值，而不用每次重新探索。講具體一點：d(C,T)=4 只算一次，之後每一條經過 C 的路徑都直接拿來用（這裡其實只有一條，S→C）——它永遠不會被重算。這個「重複利用」就是 DP 的精髓：一個節點到 T 的最便宜成本不會因為你怎麼走到它而改變，所以存一次再查表，比每條路徑都重算一遍划算多了。
