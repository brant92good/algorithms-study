Multistage Graph Shortest Path

Find the cheapest route across a graph whose nodes come in ordered stages — a problem where grabbing the cheapest first edge gets you burned, and the clean first taste of dynamic programming.

📄 Source: Alg07 · Ch 7 §1–5 (Multistage Graph; backward & forward reasoning; principle of optimality), slide 5.

🧠 Solved by: Dynamic Programming

The problem

A multistage graph is a directed graph whose nodes are partitioned into ordered groups called stages. Every edge points from one stage to the very next stage — you can never skip a stage or go backward. The first stage is a single source node S; the last stage is a single sink node T. Each edge carries a cost (its weight).

What's asked: find the minimum-cost path from S to T — the sequence of edges, one per stage transition, whose weights add up to the smallest total.

The concrete instance (our running example). The graph has 4 stages: S → then {A, B, C} → then {D, E, F} → then T. So a path is three decisions in a row: which stage-2 node, which stage-3 node, then on to T. Every edge weight:

  • Stage 1→2: S→A=1, S→B=2, S→C=5
  • Stage 2→3: A→D=4, A→E=11, B→D=9, B→E=5, B→F=16, C→F=2
  • Stage 3→4: D→T=18, E→T=13, F→T=2

Why is this hard? Why not just be greedy? The greedy method takes the cheapest edge visible right now. From S that's S→A (cost 1), then A→D (cost 4), then the forced D→T (cost 18) — total 23. But the true cheapest route is S→C→F→T = 5+2+2 = 9. The first edge S→C looks expensive (cost 5), yet it unlocks a dirt-cheap finish. A locally cheap choice can trap you in an expensive future — greedy can't see that far ahead, so we need a method that reasons about the whole path.

Watch it solved

The solving idea is backward reasoning. Let d(X,T) = the cost of the cheapest path from node X to the sink T. Start at the nodes nearest T (their answer is just the single edge to T), then step one stage back at a time: for each node take "edge to a neighbor + that neighbor's already-known d(·,T), keep the smallest." The graph is below with every weight shown; we fill each node's d(·,T) badge working backward from T. Green = its d(·,T) is known; amber = the node being computed this step. At the end the optimal path edges turn solid.

Method: Dynamic Programming Time: O(V + E) Answer: d(S,T) = 9 Beats greedy: 9 vs 23

The other direction — forward reasoning

The animation used backward reasoning: d(X,T) = cheapest cost from X to the sink, filled in from T back toward S — giving d(C,T)=4, d(A,T)=22, d(B,T)=18, and finally d(S,T)=9. The very same graph can be solved the opposite way. In forward reasoning you define d(S,X) = cheapest cost from the source S down to node X, and fill it in from S forward toward T. The stage-2 nodes are just their single edge from S (d(S,A)=1, d(S,B)=2, d(S,C)=5); then for each later node you take "each incoming neighbor's d(S,·) plus the edge into this node, keep the smallest." That gives d(S,D)=5, d(S,E)=7, d(S,F)=7, and finally d(S,T)=min{5+18, 7+13, 7+2}=9 — the same optimum of 9, reached from the front instead of the back.

Principle of optimality. Both directions work for the same reason: if a sequence of decisions is optimal overall, then any tail (or head) of that sequence must itself be optimal — so an optimal path is built from optimal sub-paths, which is exactly what lets us reuse the stored d(·) values instead of re-exploring.
