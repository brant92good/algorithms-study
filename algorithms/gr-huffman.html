Huffman Coding

Squeeze a message smaller by giving frequent symbols short codes and rare symbols long ones — built by always merging the two rarest trees.

What is it?

Computers store text as codewords — strings of bits (0s and 1s). The simplest scheme gives every symbol the same length (e.g. 3 bits each). Huffman coding does better: it builds a variable-length code where common symbols get fewer bits, so the whole message shrinks.

The catch is decoding. If A=0 and B=01, then reading 01 is ambiguous — is it B, or A then something? Huffman avoids this by producing a prefix code: no codeword is a prefix of (the start of) another. With a prefix code you can read bits left to right and the instant they match a codeword, you know that's the symbol — no lookahead, no ambiguity.

The construction is greedy. Treat each symbol as a tiny tree weighted by its frequency (how often it appears). The rule, repeated until one tree is left: take the two trees with the smallest frequencies and merge them under a new parent whose frequency is their sum. Rare symbols get merged early, so they sit deeper in the final tree and end up with longer codes. Reading the path from the root, left = 0, right = 1, spells each symbol's codeword.

Tiny example: frequencies A:5, B:2, C:1, D:1, E:1 (10 symbols total). Watch the forest of single nodes get merged, two-rarest-first, into one tree.
