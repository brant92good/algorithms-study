Branch-and-Bound

Search a tree of partial solutions best-first, but for an optimization problem — and use a lower bound at every node to throw away whole branches that cannot possibly beat the best solution you've already found.

📄 Source: A05 · The Searching Strategies §7 (branch-and-bound), §8–10 (the three problems it solves).

The idea

Plain tree searches — depth-first, breadth-first, hill climbing, best-first — all stop at the first goal node they reach. That's fine for a decision problem ("does a solution exist?"), but it's useless for an optimization problem ("find the cheapest solution"): the first goal found is almost never the best one. Branch-and-bound is the fix.

It has two ingredients, hence the name:

  • Branch — split the solution space into subsets, each a subtree. (For a tour: "tours that use arc 4→6" vs "tours that don't." For knapsack: "take item i" vs "leave it.") This is ordinary tree search.
  • Bound — for every node, compute a lower bound: a number that is guaranteed to be ≤ the cost of any complete solution living in that subtree. It's an optimistic estimate — the best the subtree could conceivably do.

You expand nodes best-first (always open the live node with the smallest lower bound, using the Step 1–4 best-first scheme). You also keep the best complete solution found so far — call its cost best. Then the whole payoff is one rule:

The prune rule. If a node's lower bound ≥ best, kill the entire subtree. Reason: every solution down there costs at least the bound, which is already no better than a solution you already have in hand — so nothing in that subtree can win. No need to look.

Why a lower bound (and why ≥, not >)? Because we're minimizing. The bound is a promise: "you can't do better than this here." If that promise is already as bad as your current champion, the branch is hopeless. (Maximization problems flip the signs — the slides actually turn the knapsack into a minimization by minimizing negative profit, so the same lower-bound machinery applies.)

A common way to get the lower bound is a relaxation: solve an easier version whose answer can only be optimistic. For assignment and TSP that's the reduced cost matrix (subtract a constant from each row/column until each has a zero; the total subtracted can't be avoided, so it's a lower bound). For knapsack it's the fractional knapsack (allow taking part of an item), solved greedily.

Complexity: worst case still exponential — these are NP-complete problems (TSP, knapsack) and B&B does not change that. Its value is pruning: a good bound chops away enormous subtrees, so in practice it explores far fewer nodes than brute force. A tighter bound prunes more.

Each tree node's number is its lower bound (cost so far + cheapest possible finish). Problems it solves (in the slides)

  • Personnel assignment — assign each person to one job at minimum total cost, while respecting a partial order on jobs (some jobs must come before others). Lower bound = reduced-cost-matrix total.
  • Travelling salesperson (TSP) — find the cheapest tour visiting all cities exactly once and returning home. Branch on including/excluding an arc; lower bound from row/column reductions. (NP-complete; optimal tour in the deck costs 126.)
  • 0/1 knapsack — pick a subset of items to maximize profit without exceeding the weight capacity. Lower bound from the fractional (greedy) relaxation; branch on take/leave each item.
