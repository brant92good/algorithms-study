Depth-First Search

Search a tree of candidate solutions by diving as deep as possible down one branch, then backtracking the moment you hit a dead end — driven by a single stack.

📄 Source: Alg05 · The Searching Strategies §1–3 (solution-space trees → DFS guided by a stack).

The idea

Many hard yes/no problems share one shape: "can I build a complete object by making a sequence of choices?" Lay those choices out as a solution tree (also called a state-space tree). The root is the empty partial solution; each edge fixes one more decision; each node is the partial solution so far; a leaf is a complete candidate. A node we are happy with is a goal node. The whole job is to find a goal node somewhere in this tree.

Depth-first search is one order for visiting that tree. The rule is simple: from the current node, walk down to its first child, then that node's first child, and keep going as deep as you can. When you reach a node that is a dead end — it violates a constraint, or it has no unexplored children — you backtrack: step back up to the most recent node that still has an untried child, and dive again. You only ever back up the minimum needed.

The bookkeeping is a stack (LIFO — last in, first out):

  1. Push the root onto the stack.
  2. Pop a node. If it is a goal node, stop — success.
  3. Otherwise push its children, and repeat from step 2 until the stack is empty (failure).

Because the stack hands back the most recently pushed node first, the children you just added are explored before their cousins — that LIFO ordering is what produces the deep-first exploration order. (Swap the stack for a FIFO queue and the very same loop becomes breadth-first search, which sweeps level by level instead.)

Why it is cheap. DFS does not need the whole tree in memory — only the current root-to-node path sits on the stack. And it can prune: the instant a partial solution already breaks a rule (a clause is false, a running sum overshoots the target), the entire subtree below it is abandoned without ever being built. That is why search beats brute-forcing all 2ⁿ candidates.

A tiny illustration

Here is the deck's sum-of-subsets tree (Fig. 6-11): from S = {7, 5, 1, 2, 10}, is there a subset summing to 9? Each node holds the running sum; going down the left edge of level i means "include element i". DFS starts at 0, includes 7, then dives. × marks a pruned dead end (sum already > 9).

0 (root, take nothing yet) └─ +7 ──▶ 7 ├─ +5 ──▶ 12 × overshoot, backtrack ├─ +1 ──▶ 8 │ ├─ +2 ──▶ 10 × overshoot │ └─ +10 ─▶ 18 × overshoot └─ +2 ──▶ 9 ◀ GOAL: {7, 2}

DFS dives 0 → 7 → 12, finds 12 > 9, backtracks to 7, tries +1 → 8, dives again to 10 and 18 (both overshoot, backtrack), pops back to 7, tries +2 → 9 — a goal. Answer: {7, 2}. The same machine never had to look at element 10 on the winning path.

One strategy, many trees. DFS itself is problem-agnostic — change only what the tree's nodes and "dead end" mean, and the identical dive-and-backtrack stack solves a whole family: SAT (branch on each variable true/false), Hamiltonian circuits (branch on the next vertex), sum-of-subsets (branch on in/out of each element). Same search, different solution trees.

Problems it solves (in the slides)

  • Satisfiability (SAT) — given a Boolean formula in n variables, is there a true/false assignment making it true? (binary tree of 2ⁿ assignments)
  • Hamiltonian Circuit — does a graph contain a cycle that visits every vertex exactly once and returns to the start?
  • Sum-of-Subsets — given a set of numbers and a target, is there a subset that sums exactly to the target?
Data structure: stack (LIFO) Memory: O(tree depth) Stops at: first goal node
