Dynamic ProgrammingDynamic Programming

Solve a problem as a sequence of decisions, then beat the brute-force blow-up by storing and reusing the optima of subproblems you'd otherwise recompute. The principle of optimality is what licenses the recurrence.把一個問題看成一連串的決定，再靠把子問題的最佳解存起來、重複利用來打敗暴力法的爆炸成長——不然你會一直重算同樣的東西。principle of optimality 就是讓你能寫出遞迴式的依據。

The idea核心概念

Dynamic programming (DP) applies when the answer to a problem can be seen as the outcome of a sequence of decisions D1, D2, …, Dn — pick the first vertex of a path, decide whether to take the first object, choose the root of a tree, and so on. A brute-force search over all decision sequences is exponential. DP cuts that down by exploiting one fact: the same subproblems reappear over and over inside that search. Solve each subproblem once, write its optimal value into a table, and reuse it instead of recomputing it. Overlapping subproblems are exactly where the savings come from.Dynamic programming (DP) 適用於：一個問題的答案可以看成一連串決定 D1, D2, …, Dn 的結果——挑路徑的第一個頂點、決定第一個物品拿不拿、選一棵樹的 root，諸如此類。把所有決定序列暴力搜一遍是 exponential 的。DP 之所以能砍掉它，靠的是一個事實：同樣的 subproblem 會在搜尋裡一再出現。每個子問題只解一次，把最佳值寫進表裡，之後重複利用而不是重算。省下來的成本，正是來自這些重疊的子問題。

Key intuition — the Principle of Optimality. If a sequence of decisions D1…Dn is optimal, then for any k the last k decisions must themselves be optimal (for the subproblem they face). If a suffix weren't optimal, swap in a better one and the whole solution improves — contradicting that it was optimal. This is the license to write a recurrence: an optimum is built out of the optima of smaller subproblems.關鍵直覺——Principle of Optimality。如果一連串決定 D1…Dn 是最佳的，那麼對任何 k，最後 k 個決定本身也一定是最佳的（針對它們面對的子問題）。反證一下：如果某個尾段不是最佳的，那把它換成更好的，整體解就會變更好——這就跟「它是最佳的」矛盾了。這正是我們能寫出遞迴式的依據：一個最佳解是由更小子問題的最佳解組成的。

So the whole method reduces to one of two equivalent moves:所以整套方法可以歸結成兩個等價的招式之一：

  1. Find the recurrence relation — express a subproblem's optimum in terms of smaller ones. For example, the 0/1 knapsack has such a recurrence, gi(y) = max{ g_{i+1}(y), g_{i+1}(y−w_{i+1}) + p_{i+1} }, where gi(y) is the best profit from objects i…n within remaining capacity y — the max just compares "skip object i" against "take it (pay weight w_{i+1}, gain profit p_{i+1})". (Why the i+1 subscripts on the right when the left says i? The slides number this object as i+1 — i.e. gi(y) stands for "objects after the first i", so the next undecided object is i+1; both branches then recurse into g_{i+1}. The indices are consistent once you read it that way.) (Full derivation is on the knapsack page; don't worry about the symbols here.) Or找出遞迴關係——用更小的子問題來表達一個子問題的最佳解。舉例來說，0/1 knapsack 就有這樣一條遞迴式 gi(y) = max{ g_{i+1}(y), g_{i+1}(y−w_{i+1}) + p_{i+1} }，其中 gi(y) 是「用第 i…n 號物品、剩下容量 y 時能拿到的最佳 profit」——那個 max 只是在比「跳過物品 i」跟「拿它（付重量 w_{i+1}、賺到 profit p_{i+1}）」哪個好。（左邊是 i、右邊卻是 i+1，會不會兜不起來？其實投影片是把「正要決定的這個物品」編號成 i+1——也就是 gi(y) 代表「前 i 個之後的物品」，所以下一個還沒決定的物品就是 i+1；兩個分支接著都遞迴到 g_{i+1}。這樣讀就對得起來了。）（完整推導在 knapsack 那頁，這裡先不用糾結這些符號。）或是
  2. Model it as a multistage graph — lay the decisions out in stages and ask for the cheapest (or most profitable) path through it.把它建成 multistage graph——把決定一個個 stage 排開，然後問穿過它最便宜（或最賺）的路徑。

The textbook shows these two routes are the same thing. A recurrence can be set up forward (define a vertex's cost using vertices nearer the source) or backward (using vertices nearer the sink). One subtle but practical point: a forward formulation is solved back-to-front. Why? If each vertex's value is defined in terms of the source side, you can't fill that value until the source-side values already exist — so you actually compute the table starting from the sink end. Concretely: the forward version defines e(X) = cheapest cost to reach X from the source, via e(X) = min over predecessors P of { e(P) + cost(P→X) }. To fill e(A) you first need e(S), to fill e(D) you first need e(A) and e(B) — so even though the formula points "back toward the source," you must process vertices in source-to-sink order, and the answer pops out at e(T). The backward version flips this: define each value from the sink and process sink-to-source. (The worked example below uses that backward formulation, defining d(X,T) from the sink, so the table fills from the sink back toward S.)課本告訴你這兩條路其實是同一回事。遞迴式可以用 forward 寫（一個頂點的成本用靠近 source 的頂點來定義）或 backward 寫（用靠近 sink 的頂點來定義）。這裡有個微妙但實用的點：forward 的寫法反而要從後往前解。為什麼？因為如果每個頂點的值是用「靠 source 那側」來定義的，那你得等那側的值先算好才填得了它——所以實際上你會從 sink 那端開始填表。具體講：forward 版本把 e(X) 定成「從 source 走到 X 的最便宜成本」，用 e(X) = min（對所有前驅 P）{ e(P) + cost(P→X) }。要算 e(A) 得先有 e(S)，要算 e(D) 得先有 e(A) 和 e(B)——所以即使公式是「往 source 那邊看」，你還是得照 source→sink 的順序一個個處理，最後答案會在 e(T) 跑出來。backward 版本剛好相反：每個值都從 sink 來定義，然後照 sink→source 的順序處理。（下面的範例走的就是這條 backward 寫法，用 sink 來定義 d(X,T)，所以表格是從 sink 往 S 方向填回去。）

Contrast with greedy. The greedy method commits to the locally cheapest choice at each step and never reconsiders. That can fail: in the graph below greedy grabs the cheap first edge S→A=1 and is forced into a total of 23, while the true optimum is 9 — reached only by paying an expensive-looking S→C=5 up front. DP avoids the trap by evaluating each decision against the optimal future it leads to, not its immediate cost.跟 greedy 對比。greedy 每一步都一口咬定當下最便宜的選擇，從不回頭。這會出包：在下面這張圖裡，greedy 抓了便宜的第一條邊 S→A=1，結果被迫走到總共 23，但真正的最佳解是 9——只有先付看起來很貴的 S→C=5 才到得了。DP 避開這個陷阱的方法，是用每個決定通往的最佳未來來評估它，而不是只看當下的成本。

A small illustration: shortest path in a multistage graph小範例：multistage graph 的最短路徑

A multistage graph is a directed graph whose vertices split into ordered stages, where every edge goes from one stage to the next, and the first and last stages each hold a single vertex (source S, sink T). The example below has 4 stages: stage 1 = {S}, stage 2 = {A,B,C}, stage 3 = {D,E,F}, stage 4 = {T}. We want the min-cost S→T path. We solve it by backward reasoning: let d(X,T) be the cheapest cost from X to T, computed from the sink backward. Notice d(D,T), d(E,T), d(F,T) each get reused by several vertices in the stage-2 group {A,B,C} — that reuse is DP at work.multistage graph 是一張有向圖，頂點分成一組組有順序的 stage，每條邊都從一個 stage 走到下一個，而且第一個和最後一個 stage 各只有一個頂點（source S、sink T）。下面這個範例有 4 個 stage：stage 1 = {S}、stage 2 = {A,B,C}、stage 3 = {D,E,F}、stage 4 = {T}。我們要找最小成本的 S→T 路徑。解法用 backward reasoning：令 d(X,T) 是從 X 到 T 的最便宜成本，從 sink 往回算。注意 d(D,T)、d(E,T)、d(F,T) 各自被 stage-2 那組 {A,B,C} 裡好幾個頂點重複利用——這個重複利用就是 DP 在發揮作用。

The same derivation in words (if you can't run the stepper above): start at the sink and work back. Base cases — the vertices one edge from T: d(D,T)=18, d(E,T)=13, d(F,T)=2. Stage 3 → stage 2: d(C,T)=2+d(F,T)=2+2=4; d(A,T)=min{4+18, 11+13}=min{22,24}=22; d(B,T)=min{9+18, 5+13, 16+2}=min{27,18,18}=18. Notice the three stored values d(D,T),d(E,T),d(F,T) get reused by A,B,C instead of recomputed — that reuse is the whole point of DP. Finally d(S,T)=min{1+22, 2+18, 5+4}=min{23,20,9}=9. The greedy first edge S→A=1 would lock you into 23; the optimum instead pays the expensive-looking S→C=5. Trace back the winning choices: S→C→F→T = 5+2+2 = 9.同一套推導，用文字講一遍（如果你跑不動上面的 stepper）：從 sink 開始往回做。Base case——離 T 只有一條邊的頂點：d(D,T)=18、d(E,T)=13、d(F,T)=2。stage 3 → stage 2：d(C,T)=2+d(F,T)=2+2=4；d(A,T)=min{4+18, 11+13}=min{22,24}=22；d(B,T)=min{9+18, 5+13, 16+2}=min{27,18,18}=18。注意 d(D,T),d(E,T),d(F,T) 這三個存好的值被 A,B,C 直接拿來用、而不是重算——這個重複利用就是 DP 的重點。最後 d(S,T)=min{1+22, 2+18, 5+4}=min{23,20,9}=9。greedy 會挑第一條便宜的邊 S→A=1，結果被鎖死在 23；最佳解反而是去付看起來很貴的 S→C=5。回溯贏的那組選擇：S→C→F→T = 5+2+2 = 9。

Problems it solves (in the slides)它能解的問題（投影片裡的）

Every problem below is a sequence of decisions whose optima overlap — so each gets a DP recurrence (often drawn as a multistage graph):下面每一題都是一連串決定、而且最佳解會重疊——所以每題都有自己的 DP 遞迴式（常常畫成 multistage graph）：

  • Multistage graph — find the minimum-cost staged path from source to sink (the motivating example; d(S,T)=9).找從 source 到 sink 的最小成本分階路徑（最入門的範例；d(S,T)=9）。
  • 0/1 Knapsack (DP) — pick a 0/1 subset of objects to maximize profit within a weight capacity, via gi(y).在重量上限內挑一個 0/1 物品子集合來最大化 profit，透過 gi(y)。
  • Resource allocation — distribute m units across n projects for maximum total profit (longest path through a staged graph).把 m 個 unit 分給 n 個 project 以求總 profit 最大（分階圖上的最長路徑）。
  • Traveling salesperson (DP) — find the cheapest tour visiting every city once, using the Held–Karp recurrence g(i,S).用 Held–Karp 遞迴式 g(i,S) 找每座城市走一次的最便宜 tour。
  • Longest common subsequence — find the longest subsequence shared by two strings, filling the table L_{i,j}.填表 L_{i,j}，找出兩個字串共有的最長 subsequence。
  • Optimal binary search tree — build the BST of least expected search cost by choosing each subtree's root, via C(i,j).透過 C(i,j)，靠著選每棵 subtree 的 root，建出期望搜尋成本最低的 BST。

The DP recipe: (1) describe the answer as a sequence of decisions; (2) confirm the principle of optimality holds; (3) write the recurrence or draw the multistage graph; (4) solve the table once, reusing each subproblem; (5) trace back to recover the actual decisions. Contrast it with the greedy bucket, which commits without reconsidering — fast when it's correct, wrong when the optimum needs an expensive early step.DP 的套路：(1) 把答案描述成一連串決定；(2) 確認 principle of optimality 成立；(3) 寫出遞迴式或畫出 multistage graph；(4) 把表解一次，每個子問題重複利用；(5) 回溯把真正的決定還原出來。可以跟 greedy 那一類對比——greedy 一決定就不回頭，對的時候很快，但當最佳解需要一個昂貴的早期步驟時就會出錯。
