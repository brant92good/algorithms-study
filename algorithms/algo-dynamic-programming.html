Dynamic Programming

Solve a problem as a sequence of decisions, then beat the brute-force blow-up by storing and reusing the optima of subproblems you'd otherwise recompute. The principle of optimality is what licenses the recurrence.

📄 Source: A07 · Dynamic Programming (sequence of decisions → overlapping subproblems → principle of optimality → multistage graphs, knapsack, TSP, LCS, OBST).

The idea

Dynamic programming (DP) applies when the answer to a problem can be seen as the outcome of a sequence of decisions D₁, D₂, …, Dₙ — pick the first vertex of a path, decide whether to take the first object, choose the root of a tree, and so on. A brute-force search over all decision sequences is exponential. DP cuts that down by exploiting one fact: the same subproblems reappear over and over inside that search. Solve each subproblem once, write its optimal value into a table, and reuse it instead of recomputing it. Overlapping subproblems are exactly where the savings come from.

Key intuition — the Principle of Optimality. If a sequence of decisions D₁…Dₙ is optimal, then for any k the last k decisions must themselves be optimal (for the subproblem they face). If a suffix weren't optimal, swap in a better one and the whole solution improves — contradicting that it was optimal. This is the license to write a recurrence: an optimum is built out of the optima of smaller subproblems.

So the whole method reduces to one of two equivalent moves:

  1. Find the recurrence relation — express a subproblem's optimum in terms of smaller ones (e.g. the knapsack gᵢ(y) = max{ g_{i+1}(y), g_{i+1}(y−w_{i+1}) + p_{i+1} }), or
  2. Model it as a multistage graph — lay the decisions out in stages and ask for the cheapest (or most profitable) path through it.

The textbook shows these two routes are the same thing. A recurrence can be set up forward (a vertex's cost defined via vertices nearer the source) or backward (via vertices nearer the sink) — and, subtly, a forward formulation is solved back-to-front, and vice versa.

Contrast with greedy. The greedy method commits to the locally cheapest choice at each step and never reconsiders. That can fail: in the graph below greedy grabs the cheap first edge S→A=1 and is forced into a total of 23, while the true optimum is 9 — reached only by paying an expensive-looking S→C=5 up front. DP avoids the trap by evaluating each decision against the optimal future it leads to, not its immediate cost.

A small illustration: shortest path in a multistage graph

A multistage graph is a directed graph whose vertices split into ordered stages, where every edge goes from one stage to the next, and the first and last stages each hold a single vertex (source S, sink T). We want the min-cost S→T path. We solve it by backward reasoning: let d(X,T) be the cheapest cost from X to T, computed from the sink backward. Notice d(D,T), d(E,T), d(F,T) each get reused by several stage-2 vertices — that reuse is DP at work.

Problems it solves (in the slides)

Every problem below is a sequence of decisions whose optima overlap — so each gets a DP recurrence (often drawn as a multistage graph):

  • Multistage graph — find the minimum-cost staged path from source to sink (the motivating example; d(S,T)=9).
  • 0/1 Knapsack (DP) — pick a 0/1 subset of objects to maximize profit within a weight capacity, via gᵢ(y).
  • Resource allocation — distribute m units across n projects for maximum total profit (longest path through a staged graph).
  • Traveling salesperson (DP) — find the cheapest tour visiting every city once, using the Held–Karp recurrence g(i,S).
  • Longest common subsequence — find the longest subsequence shared by two strings, filling the table L_{i,j}.
  • Optimal binary search tree — build the BST of least expected search cost by choosing each subtree's root, via C(i,j).

The DP recipe: (1) describe the answer as a sequence of decisions; (2) confirm the principle of optimality holds; (3) write the recurrence or draw the multistage graph; (4) solve the table once, reusing each subproblem; (5) trace back to recover the actual decisions. Contrast it with the greedy bucket, which commits without reconsidering — fast when it's correct, wrong when the optimum needs an expensive early step.
