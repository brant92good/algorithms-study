Prune-and-Search

A design strategy: each round throw away a guaranteed fraction of the input, then recurse on what's left. Because the work shrinks geometrically, the first round dominates — and many problems that look like O(n log n) collapse to O(n).

📄 Source: A06 · Prune-and-Search §1 (general method) → §2–5 (selection), §6 (linear programming), §7 (1-center).

The idea

Think about how binary search works. To find a target in a sorted array, you look at the middle, decide the target is in the left half or the right half, and discard the other half forever. Each step prunes half the data and searches what remains. That is the whole strategy, abstracted:

  • Prune. In each iteration, do some O(nᵏ) work that lets you guarantee a fixed fraction f (with 0 < f < 1) of the current input can never contain the answer. Throw it away.
  • Search. Recurse on the surviving (1−f)·n elements with the same algorithm.
  • Stop. Once the survivors shrink to a constant size, solve directly in O(1).

The key requirement — and the hard part of every prune-and-search algorithm — is choosing the pruning step so that the fraction f is guaranteed every round, on any input. A pruning rule that only sometimes discards a fraction (e.g. always pivoting on the first element) can degrade to removing one element at a time, blowing the bound to O(n²).

Why it's linear (the headline result)

Let T(n) be the running time and suppose one iteration costs O(nᵏ). Then:

T(n) = T((1−f)·n) + O(nᵏ)

Unrolling, the per-round cost forms a geometric series with ratio (1−f)ᵏ < 1:

c·nᵏ · ( 1 + (1−f)ᵏ + (1−f)²ᵏ + … ) = c·nᵏ · (a constant)

The series converges, so the sum is just a constant times the first term:

T(n) = O(nᵏ)

The punchline: the whole multi-round process costs the same order as a single round. The first iteration dominates — every later round works on an exponentially smaller input, so all of them together add only a constant factor. This is why selection, 2-variable LP, and the 1-center problem all come out O(n) even though each does O(n) work per round.

Problems it solves (in the slides)

All three pick a clever pivot/median so a constant fraction (≈¼) is provably pruned each round, turning an O(n log n)-or-worse problem into linear time:

  • 🎯 Selection — k-th smallest — given n unsorted numbers and a rank k, return the value that would sit at position k if sorted (the median when k = ⌈n/2⌉). Solved in O(n) via the median-of-medians pivot.
  • 📐 2-variable linear programming — minimize y subject to n half-plane constraints y ≥ aᵢx + bᵢ; find the lowest point of their upper envelope in O(n) by pairing constraints and testing the median intersection.
  • ⭕ 1-center — smallest enclosing circle — given n planar points, find the smallest circle that covers them all, in O(n) by pairing points and pruning on median bisector slopes.

Contrast it with divide-and-conquer. Divide-and-conquer (mergesort, etc.) keeps both halves and recombines their results — that recombination is why it lands at O(n log n). Prune-and-search keeps only one piece and never recombines, so the work decays geometrically to O(n). The three problems above — selection, 2-variable LP, and 1-center — are the canonical instances.
