Optimal Binary Search Tree

Given keys you look up at different rates, build the binary search tree where the average lookup is as cheap as possible — by trying every key as the root and remembering the best subtrees.

📄 Source: Alg07 · Ch 7 §13 (Optimal Binary Search Tree), slides 33–38.

What is it?

A binary search tree (BST) stores sorted keys so you can find one by comparing: start at the root (the top node), go left if your target is smaller, right if larger, and repeat. The number of comparisons to reach a key equals its level — the root is level 1, its children level 2, and so on. So the search cost of a key is just how deep it sits.

Now suppose some keys are searched far more often than others. The expected cost of the whole tree is each key's level weighted by how often it is searched (its probability/weight). The same set of keys can be arranged into many valid BSTs, and they are not equally good: putting a frequently-searched key near the root makes most lookups short, while burying a popular key deep makes the average lookup expensive.

The optimal BST problem: given sorted keys a₁ < a₂ < … < aₙ with weights, find the arrangement with the smallest expected cost. We solve it with dynamic programming.

The core idea — "try every key as the root." Whatever the optimal tree looks like, some key aₖ sits at its root. Everything smaller than aₖ forms its left subtree; everything larger forms its right subtree — and by the principle of optimality, those subtrees must themselves be optimal. So if we write C(i,j) for the cost of the best tree over keys aᵢ…aⱼ:

C(i,j) = mini≤k≤j [ C(i, k−1) + C(k+1, j) + W(i,j) ]

Picking root aₖ costs the best left subtree C(i,k−1) plus the best right subtree C(k+1,j) plus W(i,j), the total weight of every node in this range. Why add W(i,j)? Because giving the subtree a root pushes every node in it one level deeper, so the whole range pays one extra comparison each. We try every k and keep the cheapest.

Tiny concrete example (the set used below): keys 3, 7, 9, 12. We fill a table of C(i,j) values, smallest ranges first, so that by the time we ask "what is the best tree over all four keys?", every sub-answer it needs is already sitting in the table.
