Hamiltonian CircuitHamiltonian Circuit

Given an undirected graph, does it contain a cycle that visits every vertex exactly once and returns to where it started? Search for one by growing a path and backtracking at dead ends.給你一張無向圖，問的是：裡面有沒有一條剛好把每個 vertex 都走過一次、最後又繞回起點的環？做法是一邊把路徑接長，一邊在走到死路時回溯，去找一條出來。

The problem這題在問什麼

You are given an undirected graph — a set of vertices (dots) joined by edges (lines). A Hamiltonian circuit is a closed tour that:給你一張無向圖——就是一堆 vertex（點）用 edge（線）連起來。所謂 Hamiltonian circuit 是一條封閉的環，它要：

  • starts at some vertex,從某個 vertex 出發，
  • walks along edges of the graph,只沿著圖上的 edge 走，
  • visits every vertex exactly once (no repeats, none skipped), and每個 vertex 剛好走一次（不重複、也不能漏掉），而且
  • returns to the starting vertex at the end.最後要繞回起點那個 vertex。

The Hamiltonian circuit problem asks a yes/no question: does the given graph contain such a circuit at all? It is a feasibility problem (find one valid tour, or prove none exists) — not an optimization problem, so the edges carry no weights and there is no "cheapest" tour to compute.Hamiltonian circuit problem 問的是個 yes/no 的問題：這張圖到底有沒有這種環？這是一個 feasibility（可行性）問題——找出一條合法的環，或證明根本沒有——不是最佳化問題，所以 edge 沒有權重，也沒有什麼「最便宜」的環要算。

Do not confuse it with an Euler circuit. An Euler circuit must use every edge exactly once (vertices may repeat); a Hamiltonian circuit must visit every vertex exactly once (edges may be left unused). They sound similar but are very different problems — Euler circuits have an easy rule (all vertices even degree), while deciding Hamiltonicity is NP-complete. NP-complete just means: nobody knows a method that always decides the yes/no question quickly (in time polynomial in the number of vertices), and it is among the hardest such problems — if you found a fast method for this one, you would have a fast method for all of them. So in the worst case we are stuck searching.別跟 Euler circuit 搞混。Euler circuit 是每條 edge 剛好走一次（vertex 可以重複）；Hamiltonian circuit 是每個 vertex 剛好走一次（edge 可以不用完）。名字很像，其實是兩個差很多的問題——Euler circuit 有個簡單的判斷法則（每個 vertex 的 degree 都是偶數就有），但判斷一張圖有沒有 Hamiltonian circuit 是 NP-complete。NP-complete 簡單講就是：目前沒人知道有什麼方法能「每次都很快」（時間隨 vertex 數呈多項式成長）判出這個 yes/no，而且它是這類問題裡最難的一群——只要你替它找到一個快的方法，等於替全部這類問題都找到了。所以最壞情況下，我們就是只能硬搜。

The concrete instance (Fig. 6-6). A graph on 7 vertices numbered 1–7, with these 9 edges:具體的例子（Fig. 6-6）。一張有 7 個 vertex（編號 1–7）的圖，共有這 9 條 edge：

1–2, 1–6, 1–3, 2–3, 6–5, 7–5, 7–4, 3–4, 4–5

Question: is there a circuit through all 7 vertices? We will search for one below — and find that 1 → 6 → 5 → 7 → 4 → 3 → 2 → 1 works (check: every neighbour pair is a real edge, all 7 vertices appear once, and it closes back to 1).問題：有沒有一條繞完 7 個 vertex 的環？下面我們會去找——結果發現 1 → 6 → 5 → 7 → 4 → 3 → 2 → 1 可以（檢查一下：每個相鄰的一對都是真的有 edge、7 個 vertex 各出現一次、而且最後有繞回 1）。

The tree on the right shows the same search: × marks a dead end (no unvisited neighbour, or can't close the loop) where DFS backtracks. Step through to the found circuit. Note: the tree diagram is a simplified illustration — it shows just one representative dead-end branch (1-2-3-4-5) plus the successful path (1-6-5-7-4-3-2-1), not the full set of branches DFS would actually explore and prune.我們從 root vertex 1 開始把路徑接長，一次選一個下一個要走的 vertex（這就是 Fig. 6-8 那棵樹）。圖顯示目前走到哪——綠色 node 是走過的、金色 node 是現在這個、綠色 edge 是選到的路線。右邊的樹呈現同一個搜尋過程：× 標的是死路（沒有還沒走過的鄰居，或是接不回起點），DFS 就會在那邊回溯。一步步走到找到環為止。注意：這棵樹只是簡化的示意圖——它只畫了一條代表性的死路分支（1-2-3-4-5）加上成功的那條路徑（1-6-5-7-4-3-2-1），並不是 DFS 實際會展開、修剪的所有分支。

Which neighbour comes next? When DFS sits on a vertex it must pick one unvisited neighbour to extend to, and try them in a fixed order. Here we use the order the edges are listed in Fig. 6-6 (1–2, 1–6, 1–3, …): from vertex 1 it tries the edge to 2 first, and only after that whole branch dead-ends does it back up and try the next edge, 1→6 — which is why the search explores 1→2 before 1→6. The order is a convention, not magic: a different ordering would just find a (possibly different) circuit in a different number of steps.下一個鄰居要選哪個？當 DFS 停在某個 vertex 上時，它得從還沒走過的鄰居裡挑一個接下去，而且是照固定的順序一個一個試。這裡我們用的順序就是 Fig. 6-6 裡 edge 被列出來的順序（1–2、1–6、1–3、…）：所以站在 vertex 1 時會先試到 2 的那條 edge，等整條分支都走成死路、退回來，才換下一條 1→6——這就是為什麼搜尋會先走 1→2、再走 1→6。這個順序只是個約定、沒什麼玄機：換個順序一樣會找到（可能是另一條）環，只是花的步數不同而已。

Type: feasibility (yes/no)類型：feasibility（yes/no） Search: DFS + backtracking搜尋：DFS + 回溯 Worst case: O(n!) paths最壞情況：O(n!) 條路徑 Decision: NP-complete判定問題：NP-complete

Where does the O(n!) come from? It is the cost of the backtracking search above when nothing prunes it. Fix the start vertex; then the path picks an order for the remaining n−1 vertices: about n−1 choices for the 2nd vertex, n−2 for the 3rd, and so on down to 1. Multiplying gives (n−1)! orderings to try in the worst case — i.e. O(n!) candidate paths. DFS + backtracking abandons bad paths early, so in practice it explores far fewer, but the worst case stays factorial.那個 O(n!) 是哪來的？它就是上面這個回溯搜尋在完全沒被剪枝時的成本。先固定起點，剩下的路徑等於是在替另外 n−1 個 vertex 排順序：第 2 個 vertex 大約有 n−1 種選法、第 3 個有 n−2 種、一路下去到剩 1 種。乘起來就是最壞情況要試 (n−1)! 種排法，也就是 O(n!) 條候選路徑。DFS + 回溯會早早把爛路徑砍掉，所以實際上跑的少很多，但最壞情況還是 factorial。
