Visit every city exactly once and come home — for the cheapest total. Dynamic programming turns the factorial-many tours into a table you fill subset by subset.每座城市剛好走一次再回家，而且總花費要最便宜。dynamic programming 把多到階乘等級的 tour，變成一張按 subset 一塊一塊填的表。

📄 Source: Alg07 · Ch 7 §10 (TSP by DP / Held–Karp), slides 22–25.

🧠 Solved by: Dynamic Programming (contrast Branch-and-Bound)

The problem這題在問什麼

You are given a set of cities and a cost matrix c(i,j) = the price to travel from city i to city j. The graph is directed, so going one way can cost more than coming back (and ∞ means there is no road that direction). You must find the cheapest tour: a route that starts at city 1, visits every other city exactly once, and returns to city 1. "Cheapest" means the sum of the edge costs along the route is as small as possible.給你一組 city，還有一張 cost matrix c(i,j) = 從 city i 走到 city j 的花費。這張圖是有向的，所以去程跟回程的成本可以不一樣（∞ 表示那個方向沒路）。你要找最便宜的 tour：一條從 city 1 出發、其他每座城市剛好走一次、最後回到 city 1 的路線。「最便宜」就是路上所有邊的成本加起來越小越好。

Why is this hard? With n cities there are (n−1)! possible orderings of the cities you could visit — that explodes fast (10 cities ≈ 360,000 tours; 20 cities ≈ 10¹⁷). Checking every tour separately is hopeless. We need to share work between tours that overlap.這題難在哪？有 n 座城市，城市的走訪順序就有 (n−1)! 種——這爆得超快（10 座城市 ≈ 36 萬條 tour；20 座 ≈ 10¹⁷）。一條一條 tour 分開檢查根本沒救。我們得讓重疊的 tour 之間共用計算。

The DP idea (Held–Karp). Two half-finished tours that (a) sit at the same current city and (b) still have the same set of cities left to visit have identical futures — the cheapest way to finish is the same for both. So we don't care how we arrived, only where we are and what's left. That pair is the DP state. (Naming "what's left" or "what's already visited" is the same thing: with the city list fixed, one set is just the complement of the other, so either one pins down the state.)DP 的點子（Held–Karp）。兩條走到一半的 tour，只要 (a) 停在同一個當下城市、而且 (b) 還沒走的城市集合一樣，它們的未來就完全相同——接下來最便宜的走法對兩者是一樣的。所以我們不在乎怎麼走到這裡的，只在乎現在在哪和還剩哪些沒走。這一對就是 DP state（狀態）。（講「還剩哪些沒走」或「已經走過哪些」其實是同一件事：城市清單固定下，一個集合就是另一個的補集，所以講哪一個都能鎖定同一個 state。）

Define g(i, S) = the length of the shortest path that starts at city i, visits every city in the set S exactly once, and ends back at city 1. Here i is the city you are currently standing on — it is already accounted for, so i is NOT a member of S; and S is the set of cities still left to visit before returning to 1. So g(1, {2,3,4}) means "I'm at city 1, I still owe visits to 2, 3 and 4, then home" — city 1 is the current city, outside S. The recurrence picks the next city j to step to:定義 g(i, S) = 一條從 city i 出發、把集合 S 裡每座城市剛好走一次、最後回到 city 1 的最短路徑長度。這裡的 i 是你現在站著的那座城市——它已經算過了，所以 i 不在 S 裡面；而 S 是回家之前還沒走、剩下要走的城市集合。所以 g(1, {2,3,4}) 的意思是「我人在 city 1，還欠 2、3、4 沒走，走完再回家」——city 1 是當下城市，在 S 之外。遞迴式就是挑下一個要踏去的城市 j：

g(i, S) = minj∈S { c(i,j) + g(j, S − {j}) },   with base case g(i, ∅) = c(i,1).

Reading the base case: g(i, ∅) = c(i,1) — when S is empty there is nothing left to visit, so the only thing remaining is the single hop from i straight back to city 1, which costs exactly c(i,1).base case 怎麼讀：g(i, ∅) = c(i,1)——當 S 空了，代表沒有任何城市要走了，剩下要做的就只有從 i 直接跳回 city 1 這一步，花費剛好是 c(i,1)。

The answer is g(1, {all other cities}). We build the table up by subset size: first all the |S|=0 values, then |S|=1, and so on — each level only needs values from the level below it. Why does this order always work? In the recurrence, g(i, S) only ever reads g(j, S − {j}), and S − {j} is a strictly smaller subset (one city fewer). So everything a level-k entry depends on lives at level k−1, which we already finished — no value is ever needed before it's computed.答案是 g(1, {其他所有城市})。我們按 subset 的大小一層層往上建表：先把所有 |S|=0 的值算完，再 |S|=1，依此類推——每一層只需要用到下一層的值。為什麼這個順序一定行得通？看遞迴式就知道：g(i, S) 只會去讀 g(j, S − {j})，而 S − {j} 一定是更小的 subset（少一座城市）。所以第 k 層的值依賴的東西都落在第 k−1 層，那層我們早就填好了——絕對不會發生「要用到的值還沒算」的情況。

Time: O(n²·2ⁿ) Space: O(n·2ⁿ) exponential — but far better than n!

Where does O(n²·2ⁿ) come from? Count the table entries first: a state is a (current city, subset) pair, and there are n choices of city times 2ⁿ possible subsets — so about n·2ⁿ states to fill (that's also the space bound, since we store one value per state). Filling one state runs the min over the next city j∈S, which scans up to n choices. Multiply: n·2ⁿ states × n work each = O(n²·2ⁿ). The 2ⁿ subset factor is what keeps it exponential — but it's a far cry from the (n−1)! of checking every tour.O(n²·2ⁿ) 是怎麼來的？先數表格有幾格：一個 state 是（當下城市，subset）這一對，城市有 n 種選擇、subset 有 2ⁿ 種可能——所以大約有 n·2ⁿ 個 state 要填（這也是 space 上界，因為每個 state 存一個值）。填一個 state 要對下一個城市 j∈S 跑 min，最多掃 n 種選擇。乘起來：n·2ⁿ 個 state ×每個花 n 的工 = O(n²·2ⁿ)。讓它停在 exponential 的就是那個 2ⁿ 的 subset 因子——不過這跟「每條 tour 都檢查」的 (n−1)! 比起來，已經小太多了。
