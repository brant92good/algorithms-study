Prune-and-Search

Linear-time Selection

Find the k-th smallest element of an unsorted list in O(n) time — without sorting — by choosing a provably-good pivot: the median of medians.

What is it?

The selection problem: given a bag of n numbers and a rank k, return the value that would sit in position k if the list were sorted (the k-th smallest). When k = ⌈n/2⌉ this is the median. The obvious method — sort the whole list, then read off position k — costs O(n log n). We can do better: O(n), linear time.

The idea is prune-and-search, the same trick behind binary search: each round, throw away a fixed fraction of the data and recurse on the rest. To do that we pick a pivot p and split the list into three piles:

  • S₁ — everything less than p,
  • S₂ — everything equal to p,
  • S₃ — everything greater than p.

Now we only need one pile. Compare k to the pile sizes:

  • if |S₁| ≥ k → the answer is inside S₁, recurse there with the same k;
  • else if |S₁| + |S₂| ≥ k → p is the answer, done;
  • otherwise → the answer is in S₃; recurse there looking for rank k − |S₁| − |S₂|.

The whole thing only works if the pivot reliably splits the list — a careless pivot (like always the first element) could shave off just one element and degrade to O(n²). The fix is the median of medians: break the list into groups of 5, sort each tiny group, take each group's median, then recursively take the median of those medians as the pivot. This guarantees at least about n/4 elements get pruned every round (so at most 3n/4 survive).

Tiny example. List [3,5,7,12,19], find the 2nd smallest. Sorted it's [3,5,7,12,19], so the answer is 5 — and selection finds it without ever fully sorting the longer lists it usually runs on.
