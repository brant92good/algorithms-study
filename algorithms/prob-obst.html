Optimal Binary Search TreeOptimal Binary Search Tree

You know how often each key gets looked up. Arrange them into the binary search tree where the average lookup is as cheap as possible.你已經知道每個 key 各被查多少次。把它們排成一棵 binary search tree，讓平均查一次的成本最低。

📄 Source: Alg07 · Ch 7 §13 (Optimal Binary Search Tree), slides 33–38.

🧠 Solved by: Dynamic Programming

The problem這題在問什麼

A binary search tree (BST) stores sorted keys so you can find one by comparing as you descend: start at the root (the top node), go left if your target is smaller, right if it is larger, and repeat. The number of comparisons to reach a key equals its level — the root is level 1, its children level 2, their children level 3, and so on. So the search cost of a key is exactly how deep it sits in the tree.binary search tree (BST) 把排序好的 key 存起來，讓你一邊往下走一邊比較來找：從 root（最上面那個節點）開始，目標比較小就往左、比較大就往右，一直重複。找到一個 key 要比幾次，就等於它的 level——root 是 level 1，它的小孩 level 2，再下去 level 3，依此類推。所以一個 key 的 search cost 就是它在樹裡有多深。

Now suppose some keys are looked up far more often than others. Each key aᵢ comes with a weight Pᵢ — how often it is searched. The expected cost of the whole tree is each key's level weighted by how often it is searched:現在假設某些 key 被查的次數遠多於其他的。每個 key aᵢ 帶一個 weight Pᵢ——代表它被查的頻率。整棵樹的 expected cost 就是把每個 key 的 level 用它被查的頻率加權起來：

Cost = Σᵢ Pᵢ · level(aᵢ) + Σᵢ Qᵢ · (level of gap node − 1)

That second sum matters — it is why the table values look bigger than you expect. A real search can also miss: the target isn't any stored key. Those misses end at failure nodes (the gaps), and they cost comparisons too, so the true objective the algorithm minimizes is the keys' weighted depth plus the gaps' weighted depth. Keep this full formula in mind — every number in the table below is built from it.第二項很重要——它就是為什麼下面表格的數字看起來比你預期的大。真實的查詢也可能查不到：目標不是任何一個存好的 key。這些 miss 會停在 failure 節點（也就是 gap），而且一樣要花比較次數。所以演算法真正要最小化的，是 key 的加權深度再加上 gap 的加權深度。記住這條完整公式——下面表格裡每個數字都是用它算出來的。

The same set of keys can be arranged into many valid BSTs, and they are not equally good. Putting a frequently-searched key near the root makes most lookups short; burying a popular key deep makes the average lookup expensive. For keys 3, 7, 9, 12 alone there are several shapes — a right-leaning chain 3→7→9→12, a balanced-ish tree rooted at 7, and more — each with a different expected cost.同一組 key 可以排成很多種合法的 BST，而且它們不是一樣好。把常被查的 key 放在靠近 root 的地方，大部分查詢就很短；把熱門的 key 埋很深，平均查詢就變貴。光是 key 3, 7, 9, 12 就有好幾種形狀——往右傾的鏈 3→7→9→12、以 7 為 root 比較平衡的樹，還有更多——每種的 expected cost 都不一樣。

What's given. Sorted keys a₁ < a₂ < … < aₙ, each with a search weight Pᵢ (probability/frequency that aᵢ is the target — a successful search). Also gap weights Qᵢ (probability a target falls between two stored keys, aᵢ < x < a_{i+1} — an unsuccessful search, ending at an external "failure" node). There are n+1 gaps in all, indexed Q₀…Qₙ: the two endpoints are Q₀, the gap to the left of the first key a₁ (targets smaller than every key), and Qₙ, the gap to the right of the last key aₙ (targets larger than every key); the interior Qᵢ each sit between two adjacent keys. That is why this example lists five Q's for four keys. Watch the asymmetry: an internal key at level L costs L comparisons (you compare against it and stop), but a failure node at level L costs only L−1 — you never compare against the failure node itself; you reach it precisely because the comparison at its parent (level L−1) failed and there was nowhere left to go. So a key pays for its own level; a gap pays for its parent's level.給定的東西。排序好的 key a₁ < a₂ < … < aₙ，每個帶一個 search weight Pᵢ（aᵢ 就是目標的機率／頻率——查得到的情況）。還有 gap weight Qᵢ（目標落在兩個存好的 key 之間，aᵢ < x < a_{i+1} 的機率——查不到的情況，停在外部的「failure」節點）。gap 總共有 n+1 個，編號 Q₀…Qₙ：兩個端點分別是 Q₀，第一個 key a₁ 左邊那個 gap（目標比所有 key 都小），跟 Qₙ，最後一個 key aₙ 右邊那個 gap（目標比所有 key 都大）；中間的 Qᵢ 才是夾在相鄰兩個 key 之間的。這也是為什麼這個例子明明四個 key，卻列了五個 Q。注意這裡不對稱：一個在 level L 的內部 key 要比 L 次（你跟它比一次然後停下來）；但一個在 level L 的 failure 節點只花 L−1 次——你根本不會去跟 failure 節點本身比，你會走到它，正是因為它上面那層（level L−1）的比較失敗、又沒地方可走了。所以 key 付自己這一層的錢，gap 付的是它爸爸那層的錢。

What's asked. Among all BSTs on these keys, find the one with the smallest expected cost — the optimal binary search tree.要求的是。在這些 key 能排出的所有 BST 裡，找 expected cost 最小的那一棵——也就是 optimal binary search tree。

Why this is not obvious. You cannot just put the highest-weight key at the root: that fixes the root but the two halves it creates must also be arranged optimally, and the best root for the whole tree may not be the heaviest key once the subtree costs are counted. You need to compare every possible root, and every possible root of every subtree. That is what dynamic programming does efficiently.為什麼沒那麼直覺。你不能直接把 weight 最大的 key 放 root：這雖然定下了 root，但它切出來的左右兩半也得排得最佳，而且把 subtree 的成本算進去後，整棵樹最好的 root 不一定是最重的 key。你得比較每個可能的 root，還有每棵 subtree 的每個可能 root。這正是 dynamic programming 能有效率做到的事。

The recurrence. Whatever the optimal tree looks like, some key aₖ sits at its root. Everything smaller than aₖ forms its left subtree; everything larger forms its right subtree — and by the principle of optimality, those subtrees must themselves be optimal. Writing C(i,j) for the cost of the best tree over keys aᵢ…aⱼ:遞迴式。不管最佳樹長什麼樣，總有某個 key aₖ 坐在 root。比 aₖ 小的全變成它的左 subtree；比較大的變成右 subtree——而由 principle of optimality，這兩棵 subtree 本身也一定是最佳的。用 C(i,j) 表示 key aᵢ…aⱼ 這個範圍最佳樹的成本：

C(i,j) = mini≤k≤j [ C(i, k−1) + C(k+1, j) + W(i,j) ]

Picking root aₖ costs the best left subtree C(i,k−1) plus the best right subtree C(k+1,j) plus W(i,j), the total weight of every node in this range. Why add W(i,j)? Because giving the subtree a root pushes every node in it one level deeper — and "every node" means both the keys and the failure (gap) leaves. Each gap leaf also drops one level, so its Q weight pays one extra comparison too. That is exactly why W(i,j) sums both the P's and the Q's, not just the searched keys: the whole bundle — keys and gaps alike — gets pushed down by one when the root is added. We try every k and keep the cheapest.選 aₖ 當 root，成本就是最佳左 subtree C(i,k−1) 加最佳右 subtree C(k+1,j) 再加 W(i,j)，也就是這個範圍裡所有節點的 weight 總和。為什麼要加 W(i,j)？因為幫 subtree 配一個 root，會把裡面每個節點都往下推一層——而「每個節點」指的是 key 和 failure（gap）leaf 兩種都算。每片 gap leaf 也一起往下掉一層，所以它的 Q weight 也要多付一次比較。這正是為什麼 W(i,j) 要把 P 和 Q 兩種都加進去，不是只算被查的 key：整包東西——key 和 gap 一視同仁——在加 root 時都被往下推一層。我們把每個 k 都試一遍，留最便宜的。

Tiny concrete instance (the one solved below): keys 3, 7, 9, 12, search weights P = (3, 3, 1, 1), gap weights Q = (2, 3, 1, 1, 1). We fill a table of C(i,j) values, smallest ranges first, so that by the time we ask "what is the best tree over all four keys?", every sub-answer it needs is already in the table.小例子（下面就用它解）：key 3, 7, 9, 12，search weight P = (3, 3, 1, 1)，gap weight Q = (2, 3, 1, 1, 1)。我們填一張 C(i,j) 表，先算最小的範圍，這樣等我們問到「四個 key 全部的最佳樹是哪棵？」時，它需要的每個子答案早就在表裡了。

First, the exact W(i,j) rule. For keys aᵢ…aⱼ, the range's total weight is the keys' weights plus the gap weights on both sides of them:
W(i,j) = (Pᵢ + … + Pⱼ) + (Q₍ᵢ₋₁₎ + Qᵢ + … + Qⱼ)
— that is always one more gap weight than keys, because j−i+1 keys are separated by, and bordered by, j−i+2 gaps. The easy trap: you must include Q₍ᵢ₋₁₎, the gap just to the left of the first key. Worked example, W(2,4) over keys 7,9,12: keys give P₂+P₃+P₄ = 3+1+1 = 5; gaps give Q₁+Q₂+Q₃+Q₄ = 3+1+1+1 = 6 (note Q₁, the gap left of key 7, is included). Total W(2,4) = 5 + 6 = 11.先講 W(i,j) 的精確規則。對 key aᵢ…aⱼ 來說，這個範圍的總 weight = key 的 weight，再加上它們左右兩側的 gap weight：
W(i,j) = (Pᵢ + … + Pⱼ) + (Q₍ᵢ₋₁₎ + Qᵢ + … + Qⱼ)
——gap 永遠比 key 多一個，因為 j−i+1 個 key 之間和兩端，總共夾出 j−i+2 個 gap。最容易踩雷的地方：你一定要把 Q₍ᵢ₋₁₎ 算進去，也就是第一個 key 左邊那個 gap。實際算一次，W(2,4)（key 7,9,12）：key 部分 P₂+P₃+P₄ = 3+1+1 = 5；gap 部分 Q₁+Q₂+Q₃+Q₄ = 3+1+1+1 = 6（注意 Q₁，也就是 7 左邊那個 gap，有被算進去）。合起來 W(2,4) = 5 + 6 = 11。
Now the very first table cell, decoded: why does a one-key tree "cost" 8 when its key 3 only has weight P₁ = 3? Because the cost includes its two failure gaps. The tree for key 3 alone is: one internal node 3 at level 1, with two failure leaves Q₀ and Q₁ hanging below it at level 2. Plug into the full objective: the key contributes P₁·level = 3·1 = 3; each failure leaf is at level 2 so costs 2−1 = 1 comparison, contributing Q₀·1 + Q₁·1 = 2·1 + 3·1 = 5. Total 3 + 5 = 8. And indeed W(1,1) = P₁ + Q₀ + Q₁ = 3 + 2 + 3 = 8, so C(1,1) = W(1,1) = 8 — the base case matches the cost formula exactly.接著把表格最左上角那一格拆開來看：一個 key 的樹，key 3 的 weight 才 P₁ = 3，為什麼「成本」是 8？因為成本還包含它的兩個 failure gap。只有 key 3 的樹長這樣：一個內部節點 3 在 level 1，下面掛著兩片 failure leaf Q₀ 和 Q₁，在 level 2。代進完整目標：key 的貢獻是 P₁·level = 3·1 = 3；每片 failure leaf 在 level 2，所以花 2−1 = 1 次比較，貢獻 Q₀·1 + Q₁·1 = 2·1 + 3·1 = 5。合起來 3 + 5 = 8。而 W(1,1) = P₁ + Q₀ + Q₁ = 3 + 2 + 3 = 8 剛好一樣，所以 C(1,1) = W(1,1) = 8——base case 跟成本公式完全對得起來。
How the tree is read back out (the back-trace). The table gives the cost, but the shape comes from the winning root stored at each cell (the bold root in every caption above). Start at the whole range and recurse: C(1,4) picked root 7, so 7 is the tree's root; its left subtree is the range left of 7, C(1,1), and its right subtree is the range right of 7, C(3,4). Now expand those the same way: C(1,1) is a single key, so its root is 3 (a leaf on the left). C(3,4) picked root 9, so 9 sits there; its left side C(3,2) is empty, and its right side is the single key C(4,4) = 12. Nothing left to expand — and the nodes we placed (7 at the top; 3 and 9 below it; 12 under 9) are exactly the drawn tree.怎麼把樹倒推回來（back-trace）。表格給的是成本，但樹的形狀是靠每一格存的最佳 root 來決定的（也就是上面每個 caption 裡那個粗體 root）。從整個範圍開始往下遞迴：C(1,4) 挑的 root 是 7，所以 7 就是整棵樹的 root；它的左 subtree 是 7 左邊的範圍 C(1,1)，右 subtree 是 7 右邊的範圍 C(3,4)。接著用同樣方式展開：C(1,1) 只有一個 key，root 就是 3（左邊一片 leaf）。C(3,4) 挑的 root 是 9，所以 9 放這裡；它左邊 C(3,2) 是空的，右邊是單一 key C(4,4) = 12。沒東西可展開了——而我們擺出來的節點（7 在最上面；3 和 9 在它下面；12 在 9 下面）剛好就是畫出來的那棵樹。
Closing the loop: does C(1,4) = 32 really equal the cost of the tree we drew? The final tree is root 7 (level 1); 3 and 9 at level 2; 12 at level 3. Score it with the full objective. Keys: Σ Pᵢ·level = 3·1 (key 7) + 3·2 (key 3) + 1·2 (key 9) + 1·3 (key 12) = 3+6+2+3 = 14. Failure gaps Q₀…Q₄ = (2,3,1,1,1): Q₀,Q₁ hang under key 3 (level 3, cost 2 each), Q₂ under key 9 (level 3, cost 2), Q₃,Q₄ under key 12 (level 4, cost 3 each), giving Σ Qᵢ·(level−1) = 2·2 + 3·2 + 1·2 + 1·3 + 1·3 = 4+6+2+3+3 = 18. Total 14 + 18 = 32 = C(1,4). The recurrence really did minimize the thing the problem asked for.把圈圈收回來：C(1,4) = 32 真的等於我們畫出來那棵樹的成本嗎？最終樹是 root 7（level 1）；3 和 9 在 level 2；12 在 level 3。用完整目標來算它一次。Key 部分：Σ Pᵢ·level = 3·1（key 7）+ 3·2（key 3）+ 1·2（key 9）+ 1·3（key 12）= 3+6+2+3 = 14。Failure gap Q₀…Q₄ = (2,3,1,1,1)：Q₀,Q₁ 掛在 key 3 下面（level 3，各花 2），Q₂ 在 key 9 下面（level 3，花 2），Q₃,Q₄ 在 key 12 下面（level 4，各花 3），所以 Σ Qᵢ·(level−1) = 2·2 + 3·2 + 1·2 + 1·3 + 1·3 = 4+6+2+3+3 = 18。合計 14 + 18 = 32 = C(1,4)。遞迴式真的最小化了題目要求的那個東西。

Length 1: computing C(1,2), keys 3, 7長度 1：算 C(1,2)，keys 3, 7

Two keys can form exactly two BSTs. The gaps in this range are Q₀, Q₁, Q₂.兩個 key 剛好能排成兩種 BST。這個範圍的 gap 是 Q₀, Q₁, Q₂。

Candidate A — root 3:候選 A — root 3：

3 (lvl1) / \ Q0 7 (7 at lvl2) / \ Q1 Q2 (lvl3)

key: 3·1(key3) + 3·2(key7) = 3 + 6 = 9
gap: Q0·1 + Q1·2 + Q2·2 = 2·1+3·2+1·2 = 2 + 6 + 2 = 10
total = 19

Candidate B — root 7:候選 B — root 7：

7 (lvl1) / \ 3 Q2 (3 at lvl2) / \ Q0 Q1 (lvl3)

key: 3·1(key7) + 3·2(key3) = 3 + 6 = 9
gap: Q2·1 + Q0·2 + Q1·2 = 1·1+2·2+3·2 = 1 + 4 + 6 = 11
total = 20

Why A wins. The key cost is identical (both 9). The difference is the gaps: A puts the heavy gap Q₀ (weight 2) high up at level 2 (cost 1), while B buries it at level 3 (cost 2). Keeping heavy gaps shallow is cheaper → C(1,2) = 19, root 3.為什麼 A 贏。key 成本兩邊一樣（都 9）。差別在 gap：A 把重的 gap Q₀（weight 2）放在高處 level 2（花 1），B 卻把它壓到 level 3（花 2）。把重的 gap 放淺比較便宜 → C(1,2) = 19，root 3。

Matching the formula: root 3 → C(1,0)+C(2,2)+W(1,2) = 0+7+12 = 19 ✓; root 7 → C(1,1)+C(3,2)+W(1,2) = 8+0+12 = 20. The +W(1,2)=12 is exactly "everyone in the range drops one level when the root is added"; C(2,2)=7 is the standalone cost of the {7} subtree. The formula just packages the direct count.對回公式：root 3 → C(1,0)+C(2,2)+W(1,2) = 0+7+12 = 19 ✓；root 7 → C(1,1)+C(3,2)+W(1,2) = 8+0+12 = 20。那個 +W(1,2)=12 正是「加 root 後範圍裡每個節點都掉一層」；C(2,2)=7 是 {7} 這棵 subtree 單獨算的成本。公式只是把直接數的結果打包起來。

Length 2: computing C(1,3), keys 3, 7, 9長度 2：算 C(1,3)，keys 3, 7, 9

Now three candidate roots. The formula pulls already-stored sub-answers straight from the table (W(1,3)=14):這次有三個候選 root。公式直接從表裡抓已經存好的子答案（W(1,3)=14）：

root 3 → C(1,0)+C(2,3)+14 = 0 +12+14 = 26
root 7 → C(1,1)+C(3,3)+14 = 8 + 3+14 = 25 ✓
root 9 → C(1,2)+C(4,3)+14 = 19+ 0+14 = 33

Verify the winner (root 7) by drawing it:把贏家（root 7）畫出來驗證：

7 (lvl1) / \ 3 9 (both lvl2) /| |\ Q0 Q1 Q2 Q3 (all lvl3)

key: 3·1(key7) + 3·2(key3) + 1·2(key9) = 3 + 6 + 2 = 11
gap: (Q0+Q1+Q2+Q3)·2 = (2+3+1+1)·2 = 7·2 = 14
total = 25 ✓

Why root 7 beats the others. Root 3 buries keys 7 and 9 down the right chain (9 reaches level 3) → 26. Root 9 dumps the whole heavy left side C(1,2)=19 below it → 33. Root 7 is balanced — keys 3 and 9 both stay at level 2 → cheapest, 25.為什麼 root 7 贏過其他。root 3 把 key 7、9 擠到右邊鏈下去（9 掉到 level 3）→ 26。root 9 把整個重的左半 C(1,2)=19 都壓在底下 → 33。root 7 最平衡——key 3 和 9 都停在 level 2 → 最便宜，25。

The DP win, visible here. Computing C(1,3) at root 7 just grabbed the already-stored C(1,1)=8 and C(3,3)=3 — no recomputation. Root 3 reused C(2,3)=12 (the best tree over 7,9, which itself had chosen root 7). Small ranges are solved once, then reused by every larger range — that is exactly how DP beats brute-force tree enumeration.DP 賺到的地方，在這裡看得到。算 C(1,3) 的 root 7 時，直接抓了已經存好的 C(1,1)=8 和 C(3,3)=3——完全沒重算。root 3 重用了 C(2,3)=12（7,9 的最佳樹，它自己又是選 root 7 算出來的）。小範圍只算一次，之後被每個更大範圍重複使用——這正是 DP 打敗暴力枚舉每棵樹的原因。

Time: O(n³) Space: O(n²) Builds: min-cost BST

Primary source: CLRS, Introduction to Algorithms, §15.5 "Optimal binary search trees" (dynamic-programming chapter) — the same root-splitting recurrence and the O(n³) table fill.
